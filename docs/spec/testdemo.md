## 在哪里跑（运行宿主选择）

**结论：自建网页（你自己的最薄壳子）＞ 酒馆。** 而且前期能用 Node/CLI 测试台的部分，先别急着上画面。

<aside>
🏠

**为什么不在酒馆里开发**

- **纵切测的是引擎内核，网页薄壳让引擎成为唯一变量。** 自建壳里引擎组装器天然就是上下文唯一来源（= Y6「无历史模式」本就成立），没有楼层要对抗。
- **酒馆会把一整批已划 OUT 的宿主边界对撞拖进来**：Y1 跨标签单写者 / Y6 无历史注入面 / Y7 MVU 楼层 desync / Y4 世代号 / 楼层编辑=旁路防线。为跑纵切去解决宿主 adapter 纯属无关功。
- **引擎本就是 TS/Node**，最自然的最薄宿主就是直连引擎的小网页；确定性/重放在自己掌控的壳里最好验证。
- **酒馆是「以后要适配的目标宿主」**（P0-11 adapter·广度/兼容目标），不是「现在在上面开发的平台」。
</aside>

**分两步起：**

- **M0→M3 先用 Node/CLI 测试台驱动**（无需画面，REPLAY-01 双跑恒等在终端验证最干净）。
- **M4/M5 要「用手玩一遍」时再套极简网页**（文本流 + 一个状态栏）。
- 纵切验证完、确定进生态了，酒馆 adapter 才走 P0-11 正式接。

## 什么时候跑（时间点）

直接回答你的问题：**不是「本轮 bug 全修完、功能全加完、跑八场景之前」——而是更早、更靠前。**

**精确落点（在路线图上）：**

1. 先落地当前的 schema 黄金窗口收口批（上一轮给 CC 的 ①P0-1 schema 预埋 + ③指纹补全）——纵切要压测的就是这批你打算焊死的契约。
2. **← 就在这里跑纵切（M0→M5）**，用薄版 P0-7/8/9 把窄路打通。
3. 纵切暴露的 schema 缺口直接回填黄金窗口（还没焊，免费补）。
4. 然后才是八场景全量轰炸 + 闸二剩余轮 + 重新冻结。
5. 全绿 → 解锁 P0-6。

**逐条澄清你的三个候选时点：**

- **本轮 bug 修完？** 不必全修完。纵切只需要核心确定性原语（RNG/检定/账本/快照/重放），这些已基本就绪（P0-3/4/5 done·REPLAY-01 + C2 done）。薄版 P0-7/8/9 是纵切自己搭的最小实现，不等横向把它们做完。
- **功能加完？** 恰恰相反——纵切只穿过最小动作集，新功能（effect 包 / AOHP / 反代 / NSFW / 媒介…）全 OUT。等功能加完再跑就**晚了**：那时契约已焊死，纵切发现的缺口要付迁移。
- **跑八场景之前？** 是的，这点最关键。纵切必须在八场景全量轰炸**之前/同时**跑：①它产出的场景（悔棋/检定/账本/知情）要固化成 fixture 喂进八场景；②它的核心使命就是焊死前找漏洞，必须在 P0-6 之前。

<aside>
🧭

**一句话：「（schema 黄金窗口收口批落地之后）·（八场景全量 + 焊死之前）」这个窗口 = 纵切的正确时点。不要等全部 bug 修完、全部功能加完。**

纵切和闸二不冲突，是互补：闸二第 5/6 轮（REPLAY-01 / C2）本就要 harness，纵切给它们提供真实的端到端窄路输入；纵切跑过的恒等断言可直接并进闸二证据。

</aside>

## 一句话范围

**1 主角 + 2 NPC + 1 地点 + 一把动作 + 游戏内一天（约 5~10 拍）**，端到端跑通「组装 → 叙事 → 记账 → 最小五道闸 → 结算 → 存档 → 重放逐位恒等」，全程确定性、零 mod、零 NSFW、零多人、零媒介。

## 三 · 黄金路径（端到端七段·必须全部穿过）

这条路径就是引擎主循环的最小版。每段只做「能让下一段跑起来」的最小实现：

1. **玩家输入 → prompt 组装（P0-8 薄版）**
    - 只组装：当前主角 + 在场 NPC + 地点 + 近 K 拍叙事切片。
    - 占位解析槽（user/char 双花括号宏 → 实体键）走最小实现。
    - 记账/检定/结算调用用引擎内置 prompt，**不接任何覆盖串**（焊死前先证明核心调用结构上不含越权字段）。
2. **叙事 LLM 调用**
    - 一个 adapter，一次叙事调用，出叙事文本 + `情绪键`（已预埋字段，一次出多处消费的最小验证）。
3. **记账 LLM 调用 → 提案单**
    - 强制结构化 schema 输出提案单（给钱/收钱/检定意图）。
    - **对账覆盖性最小版**：叙事里「有后果的名词/数额」必须在提案单里有对应项，漏项 → 定点重写一次 → 仍漏降级标记。
4. **最小五道闸**
    - 结构校验（Zod）+ 账面量 clamp（复用 H1）+ 单写者（一个 worklist）+ 对账覆盖性 + 知情过滤最小版。
    - **跳过**导入闸/验签/卡判型（MVP 无 mod）。
5. **结算管线（P0-7 薄版）**
    - 单写者 worklist 不动点迭代（visited 防环 + 有界轮次）。
    - 一次技能检定走 `rngFor`（锚拍号混盐·读存档头全局回滚计数器）。
    - 给钱收钱双方账本守恒落账。
    - 盐值登记进 `tick_log`。
6. **拍前快照 + 存档（P0-9 薄版）**
    - ring buffer 近 K 拍 + 关账态入口门规 + 存档头校验和。
    - 悔棋 = 回滚到拍前快照版本（含观测值表/挂起队列，哪怕 MVP 队列为空也走同一回滚单元）。
7. **重放（REPLAY-01 跑这条窄路）**
    - 重放读 `tick_log` 冻结值（盐值/路由），LLM 调用全短路读账。
    - 同 seed 双跑逐位恒等；悔棋后重放事实逐位恒等。

### 4.6 一天剧本（7 拍·喂 fixture + 当验收用例）

3. **拍 3**：林九给红姨 2 文小费 → 账本 `pc_linjiu −2` / `npc_hong +2`（验证账本守恒 + 单写者）。

5. **拍 5**：按检定结果叙事（成功→记一笔「赊账 8 文」的账；失败→无账变，仅叙事）。
6. **拍 6**：**悔棋**回到拍 4 之前，重做一次检定（验证悔棋回滚 + 重放读 `tick_log` 冻结盐值 → 同输入逐位恒等）。
7. **拍 7**：当天结束，关账存档；随后整段重放一遍验证事实逐位恒等。

## 六 · 验收 DoD（确定性 + 可玩·两条都要）

**可玩侧：**

- [ ]  能从开局连续走到「游戏内一天结束」，中途不崩、不卡死。
- [ ]  对话/给钱/检定/悔棋四个动作都能真实触发并看到结果。

**确定性侧（不可让步）：**

- [ ]  账本守恒：给钱方 − 收钱方 = 0，越界 clamp + ⚠ 播报。
- [ ]  同档同 seed 双跑逐位恒等（复用 P0-5 gate · 走 canonicalize）。
- [ ]  悔棋后重放：读 `tick_log` 冻结盐值/路由，事实逐位恒等，live 偏好改动不漂移路由。
- [ ]  存档/读档：仅关账态存，读回继续不死档。
- [ ]  全程零 `Math.random` / 零平台 `Math` 越界（复用六禁 CI · 报红即失败）。
- [ ]  秘密不泄露：未知方的心声/菜单不暴露未知状态的存在性。

## 七 · 它如何服务 P0-6 焊死决策（最关键的产出）

- 每个这条窄路真实触碰到的 schema 字段 → 标记「已被真实运行验证」。
- 玩的过程中发现的字段缺口/接口别扭 → 直接回填 [P0 补漏执行排程 · 蓝图二次增生审计 · 剩余对撞面评估](https://app.notion.com/p/P0-b744cb81941841ceb334dd6d9f461a50?pvs=21)「schema 黄金窗口」清单（姓名知识/临时容器/effect 包/AOHP option…），**在焊死前补，不在焊死后付迁移**。
- 纵切跑过的场景（悔棋/检定/账本/知情）→ 固化成 REPLAY-01 / C2 fixture，扩充闸二对撞面。
- **判据**：纵切端到端逐位恒等跑绿 + 黄金窗口无新增缺口 = P0-6 焊死的真实信心来源（比纸面审计强得多）。

## 九 · 风险与边界

- **风险一：纵切偷工破确定性。** 对策 = 窄路也跑六禁 CI + REPLAY-01，验收 DoD 的确定性侧不可让步。
- **风险二：纵切范围悄悄膨胀回横向。** 对策 = OUT 清单是硬边界，任何「顺手加一个」先问「这条窄路必须穿过它吗」，否则回 P1/P2。
- **风险三：薄版结算/组装写法和未来正式版分叉。** 对策 = 薄版只做「最小实现」不做「临时实现」——接口签名沿用已冻结的 stub（CombatResolver/findRoute/提案单/席位表），薄在覆盖面、不薄在契约。

🔒

**全局纪律（每个里程碑都适用）**

### M0 · 接通最小调用

- **目标**：1 个 provider adapter，一次叙事调用出一句话，渲染到 CLI。
- **交付**：adapter（沿用已冻结调用注册表「叙事」条目结构）；最小 prompt 组装函数（只塞主角+在场 NPC+地点+近 K 拍）；CLI 入口 `pnpm dev:slice` 跑一拍出文本。
- **纪律**：API key 走 `.env` / 存档外机密区，绝不进存档/$层/导出。
- **验收**：终端能看到叙事文本；六禁 CI / lint 全绿。

### M1 · 单拍闭环

- **目标**：组装→叙事→记账提案→最小闸→落账→渲染 一拍走通（先不检定、不存档）。
- **交付**：记账调用出结构化提案单（Zod：给钱/收钱/检定意图）；最小五道闸的「结构校验 + 单写者 worklist + 对账覆盖性最小版」；账本落账。
- **对账覆盖性最小版**：叙事中「有后果名词/数额」逐项抽取 vs 提案单比对，漏项→定点重写一次→仍漏降级标记。
- **验收**：一拍端到端跑通；提案单非法→闸拒/降级不崩。

### M2 · 检定 + 账本守恒

- **目标**：加一次 `rngFor` 技能检定 + 给钱收钱双方账对得上 + 盐值登记 `tick_log`。
- **交付**：检定走 `rngFor`（锚拍号混盐·盐源=存档头全局回滚计数器）复用判定配方表；clamp 复用 H1；盐值写 `tick_log.盐值`。
- **纪律**：检定禁裸 `Math.random`（六禁拦）；账本守恒断言「给钱方−收钱方=0」。
- **验收**：账本守恒断言绿；同 seed 双跑检定结果逐位恒等。

### M3 · 存档 + 悔棋 + 重放

- **目标**：拍前快照 ring buffer + 悔棋回滚 + REPLAY-01 跑这条窄路逐位恒等。
- **交付**：拍前快照 ring buffer（近 K 拍）；关账态入口门规 + 存档头校验和；悔棋=回滚到拍前快照版本（含观测值表/挂起队列回滚单元，MVP 队列可空但走同一单元）；REPLAY-01 `replayTick` 接这条窄路。
- **验收**：存读不死档；悔棋后重放读 `tick_log` 冻结盐值/路由、事实逐位恒等；live 偏好改动不漂移路由（N-2 同源）。

### M4 · 知情过滤最小版 + 一个秘密

- **目标**：NPC 心声 ∩ 窗最小版（主角已知 ∩ 该 NPC 已知）+ 一个秘密不泄露；起套极简网页 UI。
- **交付**：M1 心声输出端 ∩ 窗（复用 M1 照妖镜口径）；菜单/心声不暴露未知状态存在性；极简网页（文本流 + 状态栏）。
- **验收**：秘密不泄露断言绿；能在网页里手点动作看到结果。

### M5 · 玩一整天 + DoD 验收

- **目标**：5~10 拍连跑到游戏内一天结束，过本页「六 · 验收 DoD」全部条目。
- **交付**：把蔡提供的人话剧本转成纵切 fixture；把跑过的悔棋/检定/账本/知情场景固化成 REPLAY-01 / C2 fixture。
- **验收**：六 · DoD 可玩侧 + 确定性侧全绿；产出「纵切触碰到的 schema 字段清单」+「发现的黄金窗口缺口清单」回填排程。

**每轮验收 gate**：全绿（六禁/lint/对应断言）+ commit + 测试号 +（M3 起）双跑逐位恒等 hex。

## 十一 · 你这边要准备什么（非技术人员·手把手）

<aside>
🎬

**你的角色 = 产品/导演 + 验收员；CC 的角色 = 把你的人话剧本和数值口径翻成确定性代码。** 你提供「玩什么、期望什么」，CC 保证「确定性、可重放」。

</aside>

1. **写一份「人话剧本」（最重要·只有你能给）**
    - 开局：主角是谁、在哪、想干嘛（两三句）。
    - 2 个 NPC：名字、和主角什么关系、其中一个藏着什么秘密。
    - 一天里想发生的 5~10 拍：第 1 拍主角进客栈跟谁说话、第 3 拍给某人多少钱、第 5 拍做一次什么检定（比如「说服老板赊账」）、第 7 拍悔棋回到第 5 拍重做一次……
    - 不用懂代码，就像写一段微型剧本/桌游脚本。CC 照这个搭 fixture 和验收用例。
2. **决定数值口径（你拍板·CC 落字段）**
    - 主角起始有多少钱、几条属性轴（如体质/魅力）、初始值多少。
    - 那次检定的难度（容易/中等/难）——CC 翻成 DC + 骰型。
    - 秘密「被撞破」大概多难。
3. **配一个 LLM 的 key（唯一带点技术味、但能照做）**
    - 选一个你已有 key 的 provider，拿到 API key。
    - CC 会告诉你贴进哪个文件（一般是 `.env`），你只要复制粘贴、不外传、不进存档。
4. **写一份「验收剧本」（你来当玩家）**
    - 我要点哪几个动作、每个动作期望看到什么（钱减没减、检定成没成、悔棋后是不是回到原样）。
    - M5 你照这个手点一遍，对不上就告诉 CC。
5. **学会两个动作（CC 给你一行命令，你只管跑+回贴）**
    - 「跑测试」：CC 给 `pnpm test`（或类似），你执行、把绿/红结果回贴。
    - 「跑游戏」：CC 给 `pnpm dev:slice`，你执行、在终端/网页里玩。
    - 你不需要看代码。
6. **验收里程碑的判据（你只看这三样）**
    - **能跑**（不崩、能玩到一天结束）
    - **账对**（给的钱=收的钱，检定结果合理）
    - **可重放**（同样输入跑两遍结果一字不差；悔棋后重放事实一致）

<aside>
✅

**上面 1、2、4 是只有你能做的（剧本/数值/验收口径）；3、5 是照 CC 的一步步指示做的手工活；6 是你拍板能不能过关。代码都交给 CC。**

</aside>
