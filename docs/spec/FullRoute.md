# 「AI 文游人生模拟器」酒馆 → 网页迁移路线图

<aside>
🎯

**文档目的**:把这张 SillyTavern 卡(MVU + EJS 驱动的人生模拟器)迁移成一个面向所有人的网页项目,并规划好分阶段、分轻重缓急的落地路线。

**总策略**:免费静态单机先行 → 内容打磨 → 弱社交 → 社区 mod → 多人 → 付费规模化。每一步都为下一步留接口,但绝不提前建设。

</aside>

## 优先级图例

- 🔴 **必做(P0)**:本阶段地基,不做无法推进
- 🟡 **重要(P1)**:显著提升体验/质量,应尽快做
- 🟢 **可延后(P2)**:锦上添花,有余力再做

---

## 一、核心架构映射:酒馆 → 网页

酒馆为这张卡免费提供的能力,迁移后都要自己补一层:

| 酒馆提供的 | 网页要自己实现的 |
| --- | --- |
| MVU 变量引擎(getvar/setvar/Mvu.*) | 自写 stat_data 读写 + JSONPatch 应用器(含自定义 delta op) |
| 世界书条目注入(getwi/@@activate) | 自写 prompt 组装层:按游戏模式拼接对应指令块 |
| EJS 模板执行(&lt;%- %&gt;) | 把 EJS 逻辑改写成普通 JS 函数 |
| 聊天/楼层管理(getChatMessages) | 自写对话历史与回合状态管理 |
| 脚本变量持久化 + 按钮/事件(eventOn) | localStorage(后期换后端)+ 自写 UI 事件 |
| LLM 接口(酒馆代发) | 玩家自填 key,前端直连 CORS 友好的 provider |

---

## 二、贯穿全程的工程总原则(地基)

<aside>
🏗️

无论做到哪个阶段,始终保持四层清晰解耦,让未来的社交/多人改造是"可控重构"而非"推倒重来":

</aside>

1. **状态层**:`stat_data` 做成纯粹、可序列化的对象;所有读写走统一的 `getState / applyPatch` 接口。
2. **逻辑层**:调度器、JSONPatch 应用器都是**纯函数**(输入状态、输出结果),不直接操作界面 → 未来可搬到服务器端跑。
3. **存储层**:抽象 `Storage.save / load`,现在=localStorage,未来=后端 API,只改这一层。
4. **角色维度预留**:现在不实现多角色,但**别把"单主角"假设写死**到每个角落(尤其记忆召回里硬编码的 `stat_data.主角`,留参数化空间)。

> 你的事件包/NPC 包管理器已经做到了这一点:`IMPORT` 接收 JSON 文本,与"JSON 从哪来"完全解耦 —— 未来社区市场只是新的 JSON 来源。这就是正确范式。
>

## 三、分阶段路线图

### 阶段 0 · 架构定稿与准备

> **目标**:动手前把地基和替代层设计定稿,避免后期返工。
> 
- 🔴 确定技术栈(纯静态:HTML/JS + 可选轻量框架),仓库与 GitHub Pages 部署流程跑通
- 🔴 数据模型定稿:`stat_data` 序列化结构 + 存档接口 + 逻辑层纯函数边界
- 🔴 列出酒馆专属依赖清单(getvar/setvar/getwi/Mvu/triggerSlash/eventOn/getChatMessages…)→ 设计一一对应的替代层(shim)
- 🟡 **i18n 架构早做**:文案与逻辑分离(英/中/日),哪怕先只做中文,也先把多语言框架搭好
- 🟢 多角色维度的预留设计(只设计,不实现)

**退出标准**:一个空壳静态页能部署上线,架构文档清晰。

---

### 阶段 1 · 单机 MVP(免费静态,GitHub Pages)⭐ 重心

> **目标**:网页上能完整跑通一局核心循环(创建 → 日程 → 推进 → 结算 → 总结)。这是整个项目的命门。
> 
- 🔴 移植 **MVU 引擎**:stat_data 读写 + JSONPatch 应用(replace/delta/insert/remove,注意 delta 是自定义数值增减 op)
- 🔴 移植 **EJS 调度器**:游戏模式状态机(CHARACTER_CREATION/SCHEDULE/TIME_ADVANCE/EVENT_SETTLEMENT/RP_MODE/SUMMARY)
- 🔴 **LLM 接入**:玩家自填 key,前端直连(Gemini/OpenRouter/DeepSeek/Claude 等 CORS 友好的)
- 🔴 **前缀权限**落地:无前缀=AI 可读写、`_`=只读、`$`=AI 不可见(过滤后再注入 prompt)
- 🔴 基础 UI:叙事窗 + 状态栏 + 输入框;localStorage 存档
- 🟡 各模式指令移植(角色创建/日程/时间推进/事件结算/RP/总结)
- 🟡 状态栏模板(变量列表)+ 错误自救(state_sync)
- 🟢 记忆召回三系统**可后置**到阶段 2,先把主循环跑通

**退出标准**:能从角色创建一路玩到时间推进/事件结算,存档可续。

---

### 阶段 2 · 内容打磨 + 本地自定义体系

> **目标**:把"能玩"变成"好玩、稳定、可自定义"。验证核心玩法是否成立。
> 
- 🔴 移植**记忆召回三系统**(触景生情 / 记忆关联 / 彩蛋浮现 —— 已确认是纯 JS,可直接迁移)
- 🔴 接入你已做好的**事件包 / NPC 包管理器**(本地导入/校验/启用,含 `ATTR_WHITELIST` 沙箱)
- 🟡 数值平衡 + prompt 质量打磨;**导出功能**(把自制包/存档导出成 JSON 文件,方便手动分享备份)
- 🟡 包的示例模板 + 友好报错 UI
- 🟢 多存档 / 多周目 / UI 美化

<aside>
🔒

**审核说明**:本地导入是"私密内容",玩家自制包顶多坑自己,**几乎无需审核**(白名单已沙箱化数值写入)。审核压力要等到内容"公开分发"时才上升(见阶段 4)。

</aside>

**退出标准**:朋友试玩反馈"好玩",且能自己做包导入。

---

### 阶段 3 · 弱社交(首次引入轻后端)

> **目标**:性价比最高的第一步社交 —— 分享人生轨迹 + 排行,只需很轻的后端。
> 
- 🔴 引入轻后端(Supabase / Firebase)+ 账号体系
- 🔴 **分享人生轨迹**:把一局人生导出成只读快照页 + 分享链接
- 🟡 排行榜 + 人生轨迹页美化
- 🟢 点赞 / 评论

**退出标准**:玩家能分享自己的人生并浏览他人的。

---

### 阶段 4 · 社区 mod 市场

> **目标**:把"本地导入"升级成"中心化浏览/上传/下载他人的事件包与 NPC 包"。
> 
- 🔴 包的上传 / 浏览 / 下载(**复用现有 `LIFE_MODPACK_IMPORT`,只是新增"从服务器拉 JSON"的来源**)
- 🟡 评分 / 标签筛选 / 订阅
- 🟢 创作者主页

<aside>
⚠️

这是法律内容风险**真正落到你头上**的节点:你在向所有人分发他人创作的内容。审核不到位,前面讲的合规风险会在这里集中爆发。

</aside>

**退出标准**:玩家能安全地分享和获取 mod,有审核兜底。

---

### 阶段 5 · 多人(同步回合)

> **目标**:实现"提交 → 显示他人输入中 → 收齐 → 批量发 AI → 结果一并广播"的锁步同步回合。
> 
- 🔴 房间系统 + **回合协调器状态机**(等所有人提交 → 批量 prompt → 单次生成 → 广播)
- 🔴 WebSocket 实时层(presence / 回合状态 / 结果广播)
- 🔴 **房主 key 代理**(存后端,绝不下发给其他玩家)
- 🔴 **共享世界数据模型**:单主角 → 共享世界 + 多玩家角色(阶段 0 已预留)
- 🔴 **闸门健壮性**:超时兜底 / 掉线重连 / 房主强制推进(防一个挂机锁全局)
- 🟡 "正在输入中"指示;**死亡/继承处理**(复用卡内世代/血缘机制,死者转世重开,世界继续)
- 🟡 生成期间流式广播
- 🟢 4–6 人上限管理;人多时分组/分场景

**退出标准**:4–6 人能稳定玩一局同步回合,断线可恢复。

---

### 阶段 6 · 付费与规模化(合规优先)

> **目标**:商业化。但合规与个人身份风险必须先行。
> 
- 🔴 **正式做大/收费前咨询执业律师**
- 🟡 付费计划设计 + 成本控制(AI 调用是大头)
- 🟢 多语言市场扩展

<aside>
⚖️

服务器在日本**不能**免除中国法律:只要主要面向国内宣传 + 向国内收款,中国的 AI / 版号 / 内容监管照样适用。两国风险是叠加的。优先做海外/外语免费版最稳。

</aside>
